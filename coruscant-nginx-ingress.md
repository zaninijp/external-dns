  <div id="readme" class="Box-body readme blob js-code-block-container p-5 p-xl-6 gist-border-0">
    <article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-setting-up-externaldns-on-gke-with-nginx-ingress-controller" class="anchor" aria-hidden="true" href="#setting-up-externaldns-on-gke-with-nginx-ingress-controller"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Setting up ExternalDNS on GKE with nginx-ingress-controller</h1>
<p>This tutorial describes how to setup ExternalDNS for usage within a GKE cluster that doesn't make use of Google's <a href="https://github.com/kubernetes/ingress-gce">default ingress controller</a> but rather uses <a href="https://github.com/kubernetes/ingress-nginx">nginx-ingress-controller</a> for that task.</p>
<h2><a id="user-content-set-up-your-environment" class="anchor" aria-hidden="true" href="#set-up-your-environment"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Set up your environment</h2>
<p>Setup your environment to work with Google Cloud Platform. Fill in your values as needed, e.g. target project.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud config <span class="pl-c1">set</span> project <span class="pl-s"><span class="pl-pds">"</span>zalando-external-dns-test<span class="pl-pds">"</span></span></span>
$ <span class="pl-s1">gcloud config <span class="pl-c1">set</span> compute/region <span class="pl-s"><span class="pl-pds">"</span>europe-west1<span class="pl-pds">"</span></span></span>
$ <span class="pl-s1">gcloud config <span class="pl-c1">set</span> compute/zone <span class="pl-s"><span class="pl-pds">"</span>europe-west1-d<span class="pl-pds">"</span></span></span></pre></div>
<h2><a id="user-content-gke-node-scopes" class="anchor" aria-hidden="true" href="#gke-node-scopes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>GKE Node Scopes</h2>
<p>The following instructions use instance scopes to provide ExternalDNS with the
permissions it needs to manage DNS records. Note that since these permissions
are associated with the instance, all pods in the cluster will also have these
permissions. As such, this approach is not suitable for anything but testing
environments.</p>
<p>Create a GKE cluster without using the default ingress controller.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud container clusters create <span class="pl-s"><span class="pl-pds">"</span>external-dns<span class="pl-pds">"</span></span> \</span>
<span class="pl-c1">    --num-nodes 1 \</span>
<span class="pl-c1">    --scopes "https://www.googleapis.com/auth/ndev.clouddns.readwrite"</span></pre></div>
<p>Create a DNS zone which will contain the managed DNS records.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud dns managed-zones create <span class="pl-s"><span class="pl-pds">"</span>external-dns-test-gcp-zalan-do<span class="pl-pds">"</span></span> \</span>
<span class="pl-c1">    --dns-name "external-dns-test.gcp.zalan.do." \</span>
<span class="pl-c1">    --description "Automatically managed zone by ExternalDNS"</span></pre></div>
<p>Make a note of the nameservers that were assigned to your new zone.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud dns record-sets list \</span>
<span class="pl-c1">    --zone "external-dns-test-gcp-zalan-do" \</span>
<span class="pl-c1">    --name "external-dns-test.gcp.zalan.do." \</span>
<span class="pl-c1">    --type NS</span>
<span class="pl-c1">NAME                             TYPE  TTL    DATA</span>
<span class="pl-c1">external-dns-test.gcp.zalan.do.  NS    21600  ns-cloud-e1.googledomains.com.,ns-cloud-e2.googledomains.com.,ns-cloud-e3.googledomains.com.,ns-cloud-e4.googledomains.com.</span></pre></div>
<p>In this case it's <code>ns-cloud-{e1-e4}.googledomains.com.</code> but your's could slightly differ, e.g. <code>{a1-a4}</code>, <code>{b1-b4}</code> etc.</p>
<p>Tell the parent zone where to find the DNS records for this zone by adding the corresponding NS records there. Assuming the parent zone is "gcp-zalan-do" and the domain is "gcp.zalan.do" and that it's also hosted at Google we would do the following.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud dns record-sets transaction start --zone <span class="pl-s"><span class="pl-pds">"</span>gcp-zalan-do<span class="pl-pds">"</span></span></span>
$ <span class="pl-s1">gcloud dns record-sets transaction add ns-cloud-e{1..4}.googledomains.com. \</span>
<span class="pl-c1">    --name "external-dns-test.gcp.zalan.do." --ttl 300 --type NS --zone "gcp-zalan-do"</span>
$ <span class="pl-s1">gcloud dns record-sets transaction execute --zone <span class="pl-s"><span class="pl-pds">"</span>gcp-zalan-do<span class="pl-pds">"</span></span></span></pre></div>
<p>Connect your <code>kubectl</code> client to the cluster you just created and bind your GCP
user to the cluster admin role in Kubernetes.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud container clusters get-credentials <span class="pl-s"><span class="pl-pds">"</span>external-dns<span class="pl-pds">"</span></span></span>
$ <span class="pl-s1">kubectl create clusterrolebinding cluster-admin-me \</span>
<span class="pl-c1">    --clusterrole=cluster-admin --user="$(gcloud config get-value account)"</span></pre></div>
<h3><a id="user-content-deploy-the-nginx-ingress-controller" class="anchor" aria-hidden="true" href="#deploy-the-nginx-ingress-controller"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Deploy the nginx ingress controller</h3>
<p>First, you need to deploy the nginx-based ingress controller. It can be deployed in at least two modes: Leveraging a Layer 4 load balancer in front of the nginx proxies or directly targeting pods with hostPorts on your worker nodes. ExternalDNS doesn't really care and supports both modes.</p>
<h4><a id="user-content-default-backend" class="anchor" aria-hidden="true" href="#default-backend"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Default Backend</h4>
<p>The nginx controller uses a default backend that it serves when no Ingress rule matches. This is a separate Service that can be picked by you. We'll use the default backend that's used by other ingress controllers for that matter. Apply the following manifests to your cluster to deploy the default backend.</p>
<div class="highlight highlight-source-yaml"><pre><span class="pl-ent">apiVersion</span>: <span class="pl-c1">v1</span>
<span class="pl-ent">kind</span>: <span class="pl-s">Service</span>
<span class="pl-ent">metadata</span>:
  <span class="pl-ent">name</span>: <span class="pl-s">default-http-backend</span>
<span class="pl-ent">spec</span>:
  <span class="pl-ent">ports</span>:
  - <span class="pl-ent">port</span>: <span class="pl-c1">80</span>
    <span class="pl-ent">targetPort</span>: <span class="pl-c1">8080</span>
  <span class="pl-ent">selector</span>:
    <span class="pl-ent">app</span>: <span class="pl-s">default-http-backend</span>

---

<span class="pl-ent">apiVersion</span>: <span class="pl-s">apps/v1</span>
<span class="pl-ent">kind</span>: <span class="pl-s">Deployment</span>
<span class="pl-ent">metadata</span>:
  <span class="pl-ent">name</span>: <span class="pl-s">default-http-backend</span>
<span class="pl-ent">spec</span>:
  <span class="pl-ent">selector</span>:
    <span class="pl-ent">matchLabels</span>:
      <span class="pl-ent">app</span>: <span class="pl-s">default-http-backend</span>
  <span class="pl-ent">template</span>:
    <span class="pl-ent">metadata</span>:
      <span class="pl-ent">labels</span>:
        <span class="pl-ent">app</span>: <span class="pl-s">default-http-backend</span>

<p>By default, the controller will update your Ingress objects with the public IPs of the nodes running your nginx controller instances. You should run multiple instances in case of pod or node failure. The controller will do leader election and will put multiple IPs as targets in your Ingress objects in that case. It could also make sense to run it as a DaemonSet. However, we'll just run a single replica. You have to open the respective ports on all of your worker nodes to allow nginx to receive traffic.</p>

<p>The following instructions use <a href="https://cloud.google.com/kubernetes-engine/docs/how-to/workload-identity" rel="nofollow">GKE workload
identity</a>
to provide ExternalDNS with the permissions it needs to manage DNS records.
Workload identity is the Google-recommended way to provide GKE workloads access
to GCP APIs.</p>
<p>Create a GKE cluster with workload identity enabled and without the
HttpLoadBalancing add-on.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud container clusters create external-dns \</span>
<span class="pl-c1">    --workload-metadata-from-node=GKE_METADATA_SERVER \</span>
<span class="pl-c1">    --identity-namespace=zalando-external-dns-test.svc.id.goog \</span>
<span class="pl-c1">    --addons=HorizontalPodAutoscaling</span></pre></div>
<p>Create a GCP service account (GSA) for ExternalDNS and save its email address.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">sa_name=<span class="pl-s"><span class="pl-pds">"</span>Kubernetes external-dns<span class="pl-pds">"</span></span></span>
$ <span class="pl-s1">gcloud iam service-accounts create sa-edns --display-name=<span class="pl-s"><span class="pl-pds">"</span><span class="pl-smi">$sa_name</span><span class="pl-pds">"</span></span></span>
$ <span class="pl-s1">sa_email=<span class="pl-s"><span class="pl-pds">$(</span>gcloud iam service-accounts list --format=<span class="pl-s"><span class="pl-pds">'</span>value(email)<span class="pl-pds">'</span></span> \</span></span>
<span class="pl-c1">    --filter="displayName:$sa_name")</span></pre></div>
<p>Bind the ExternalDNS GSA to the DNS admin role.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud projects add-iam-policy-binding zalando-external-dns-test \</span>
<span class="pl-c1">    --member="serviceAccount:$sa_email" --role=roles/dns.admin</span></pre></div>
<p>Link the ExternalDNS GSA to the Kubernetes service account (KSA) that
external-dns will run under, i.e., the external-dns KSA in the external-dns
namespaces.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud iam service-accounts add-iam-policy-binding <span class="pl-s"><span class="pl-pds">"</span><span class="pl-smi">$sa_email</span><span class="pl-pds">"</span></span> \</span>
<span class="pl-c1">    --member="serviceAccount:zalando-external-dns-test.svc.id.goog[external-dns/external-dns]" \</span>
<span class="pl-c1">    --role=roles/iam.workloadIdentityUser</span></pre></div>
<p>Create a DNS zone which will contain the managed DNS records.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud dns managed-zones create external-dns-test-gcp-zalan-do \</span>
<span class="pl-c1">    --dns-name=external-dns-test.gcp.zalan.do. \</span>
<span class="pl-c1">    --description="Automatically managed zone by ExternalDNS"</span></pre></div>
<p>Make a note of the nameservers that were assigned to your new zone.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud dns record-sets list \</span>
<span class="pl-c1">    --zone=external-dns-test-gcp-zalan-do \</span>
<span class="pl-c1">    --name=external-dns-test.gcp.zalan.do. \</span>
<span class="pl-c1">    --type NS</span>
<span class="pl-c1">NAME                             TYPE  TTL    DATA</span>
<span class="pl-c1">external-dns-test.gcp.zalan.do.  NS    21600  ns-cloud-e1.googledomains.com.,ns-cloud-e2.googledomains.com.,ns-cloud-e3.googledomains.com.,ns-cloud-e4.googledomains.com.</span></pre></div>
<p>In this case it's <code>ns-cloud-{e1-e4}.googledomains.com.</code> but your's could
slightly differ, e.g. <code>{a1-a4}</code>, <code>{b1-b4}</code> etc.</p>
<p>Tell the parent zone where to find the DNS records for this zone by adding the
corresponding NS records there. Assuming the parent zone is "gcp-zalan-do" and
the domain is "gcp.zalan.do" and that it's also hosted at Google we would do the
following.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud dns record-sets transaction start --zone=gcp-zalan-do</span>
$ <span class="pl-s1">gcloud dns record-sets transaction add ns-cloud-e{1..4}.googledomains.com. \</span>
<span class="pl-c1">    --name=external-dns-test.gcp.zalan.do. --ttl 300 --type NS --zone=gcp-zalan-do</span>
$ <span class="pl-s1">gcloud dns record-sets transaction execute --zone=gcp-zalan-do</span></pre></div>
<p>Connect your <code>kubectl</code> client to the cluster you just created and bind your GCP
user to the cluster admin role in Kubernetes.</p>
<div class="highlight highlight-text-shell-session"><pre>$ <span class="pl-s1">gcloud container clusters get-credentials external-dns</span>
$ <span class="pl-s1">kubectl create clusterrolebinding cluster-admin-me \</span>
<span class="pl-c1">    --clusterrole=cluster-admin --user="$(gcloud config get-value account)"</span></pre></div>
